# 仕様管理フロー

## 原則

ユーザーが口を挟む = 期待と乖離している = 仕様の不足

## フロー

```
介入 → プランモード → Issue → 実装
```

1. ユーザー介入を検知
2. プランモードで計画立案
3. GitHub Issueに記録
4. 実装・コミット（Issue参照）

## なぜIssueか

コンテキスト喪失に備える唯一の手段。

## 例外なし

即対応できても、Issueは立てる。

## 仕様不整合発見時

ユーザー指示と既存仕様（README等）が矛盾する場合：

1. **確認**: どちらが正しいか聞く
2. **勝手に選ばない**: 片方を独断で採用しない
3. **両方更新**: コードとドキュメントを正しい方に合わせる

## 教訓もIssue化

対応で得た教訓・改善点もIssueに記録する。
