# AGENTS.md / Non-negotiables の書き方ガイド

AIエージェントへのルール・哲学・制限を明文化する。

## AGENTS.md の構成

```markdown
# AGENTS.md

## プロジェクト概要
[1-2文でプロジェクトの目的]

## Non-negotiables（絶対ルール）
[破ったら即リジェクトするルール]

## ワークストリーム
[担当領域の分離]

## 哲学・原則
[判断に迷った時の指針]

## リソース制限
[タイムアウト、メモリ、並列数]

## テスト組織
[テストの配置ルール]

## 禁止事項
[やってはいけないこと]

## 推奨事項
[やるべきこと]
```

## Non-negotiables の書き方

**具体的・検証可能・例外なし** で書く。

```markdown
## Non-negotiables

以下のルールは例外なく遵守すること。違反したコードはマージしない。

### コード品質
- [ ] 本番コードで `panic!()` / `unwrap()` 禁止（Result型で処理）
- [ ] 未使用のimport/変数を残さない
- [ ] TODO/FIXMEコメントにはIssue番号を付ける

### セキュリティ
- [ ] APIキー・認証情報をコードに含めない
- [ ] ユーザー入力は必ずバリデーション
- [ ] SQLは必ずプリペアドステートメント

### 互換性
- [ ] 特定サイト向けのハックコード禁止
- [ ] 仕様に違反する実装禁止
- [ ] 処理パイプラインの順序変更禁止

### プロンプト（AI機能）
- [ ] 推測の場合は必ず「推測」と明示
- [ ] 判断根拠がある場合は必ず参照
- [ ] ユーザーの質問に直接回答
```

## ワークストリーム定義

```markdown
## ワークストリーム

| 領域 | 担当 | 主要ファイル |
|------|------|-------------|
| UI | Agent-A | components/, pages/ |
| API | Agent-B | services/, api/ |
| DB | Agent-C | models/, migrations/ |
| Test | Agent-D | tests/, __tests__/ |

### 所有権ルール
- 各ワークストリームのファイルは担当エージェントのみが編集
- 共有インターフェース（types/）は Agent-A が管理
- 変更が複数領域にまたがる場合は事前に調整
```

## 哲学・原則

```markdown
## 哲学

### 判断に迷ったら
1. ユーザーの利益を最優先
2. シンプルな方を選ぶ
3. 後から変更しやすい方を選ぶ
4. 分からなければ聞く

### コード哲学
- 動くコード > 完璧なコード
- 読みやすさ > 短さ
- 明示的 > 暗黙的
- 早期リターン > ネスト

### エラー処理哲学
- 失敗は想定内
- エラーメッセージは具体的に
- リカバリー可能なら回復を試みる
- 無理なら早めに失敗
```

## リソース制限

```markdown
## リソース制限

### タイムアウト
- 単体テスト: 10秒
- 統合テスト: 60秒
- ビルド: 5分
- AI API呼び出し: 30秒

### メモリ
- テスト実行: 最大2GB
- ビルド: 最大4GB

### 並列数
- テスト: 4並列
- AIエージェント: 2並列
```
