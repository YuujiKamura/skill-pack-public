# タスク粒度判定・分割スキル

## 判定基準

タスクが以下に該当する場合、分割が必要:

1. **3ファイル以上を変更する** → ファイルグループごとに分割
2. **2モジュール以上にまたがる** → モジュールごとに分割
3. **独立した変更が混在** → 並列実行可能な単位に分割
4. **推定作業時間が長い**（型変更の波及、大規模リファクタ等） → フェーズに分割

## 分割ルール

### ファイル競合回避
- 同じファイルを触るタスクは同一エージェントに割り当てるか、依存関係を設定
- types.rs, lib.rs, mod.rs など共有ファイルは「最後に統合」タスクで処理

### 依存関係の検出
- 型定義の移動 → 参照元の更新は同一タスクまたはブロック依存
- エラー型変更 → 全モジュールに波及するので最後に実施
- mod.rs の変更 → モジュール追加/削除と同時に実施

### 並列化の原則
- 異なるモジュール（export/, hierarchy/, analyzer/）は並列可
- 同一ディレクトリ内でも異なるファイルなら並列可
- 共有ファイル（types.rs, error.rs, lib.rs）を触るタスクは直列

## チーム構成パターン

### 小規模（3タスク以下）
- チーム不要、順次実行

### 中規模（4-8タスク）
- 2-3エージェント並列
- リーダーが統合・ビルド確認

### 大規模（9タスク以上）
- フェーズ分割（Phase A: 独立タスク並列、Phase B: 依存タスク並列、Phase C: 統合）
- 5エージェント上限（それ以上はファイル競合リスク増大）
- 実績: photo-ai-rust arch-fix-round2（10タスク、5エージェント並列→3フェーズ完了）

### コンテキスト喪失対策
- セッション切れ後はタスクリスト+git status+git diffで状態復元
- エージェントの未コミット変更はリーダーが引き継いでコミット
- ビルド+テスト通過を確認してから次フェーズへ

## タスク記述テンプレート

```
件名: [モジュール名]: [変更内容の要約]
説明:
- 変更対象ファイル（明示的にリスト）
- 変更内容（番号付き手順）
- 注意事項（競合ファイル、依存関係）
- 検証方法（cargo build, cargo test）
```

## 分割の実行手順

1. レビュー指摘またはタスク一覧を受け取る
2. 各指摘の変更対象ファイルをリストアップ
3. ファイル依存グラフを作成（共有ファイルを特定）
4. 独立グループに分割
5. グループ内で並列可能なタスクを特定
6. 依存関係（blockedBy）を設定
7. TaskCreateで登録、エージェントを投入
