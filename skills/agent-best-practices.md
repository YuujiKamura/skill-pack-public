## 1. エージェントハーネスの理解

エージェントの性能は**3つの要素**で決まる：

| 要素 | 説明 | 例 |
|------|------|-----|
| **Instructions** | エージェントの振る舞いを規定するシステムプロンプト | CLAUDE.md, .cursor/rules/ |
| **Tools** | ファイル編集、コード検索、ターミナル実行など | Edit, Grep, Bash |
| **User messages** | ユーザーからのプロンプトとフォローアップ | 具体的な指示、質問 |

**ポイント**: モデルごとに最適化されたハーネスが重要。開発者はソフトウェア開発に集中できる。

## 2. プランニング優先アプローチ

### なぜプランニングが重要か

> シカゴ大学の研究: 経験豊富な開発者はコーディング前に計画を立てる

### Plan Mode の活用

**起動方法**: `Shift+Tab` (Cursor) または EnterPlanMode ツール (Claude Code)

**Plan Mode でエージェントができること**:
1. コードベースを調査して関連ファイルを特定
2. 明確化のための質問
3. ファイルパスを含む詳細な実装計画の作成
4. 承認を待ってから実行

### 計画の保存と活用

- 計画は `.cursor/plans/` に保存される
- チームドキュメントとして機能
- 将来のエージェント作業のコンテキストになる

### プランニングのベストプラクティス

```markdown
## 良いプランの例

### 目的
ユーザー認証機能の追加

### 調査結果
- 既存の認証: なし
- 使用フレームワーク: Express.js
- 関連ファイル: src/routes/, src/middleware/

### 実装計画
1. src/middleware/auth.ts - JWT検証ミドルウェア作成
2. src/routes/auth.ts - ログイン/ログアウトエンドポイント
3. src/models/user.ts - ユーザーモデル更新
4. tests/auth.test.ts - テスト作成

### 確認事項
- [ ] JWTの有効期限は？
- [ ] リフレッシュトークンは必要か？
```

## 3. コンテキスト管理

### 検索の委任

エージェントは `grep` とセマンティック検索を使って関連ファイルを自動的に見つける。

**NG**: 関係のないファイルを手動でタグ付け
**OK**: エージェントに検索を任せる

### 会話のライフサイクル

| 状況 | アクション |
|------|----------|
| タスクを切り替える | **新しい会話を開始** |
| エージェントが混乱している | **新しい会話を開始** |
| 論理的な単位が完了した | **新しい会話を開始** |
| 機能を反復開発中 | **会話を継続** |
| 直近の作業をデバッグ中 | **会話を継続** |
| 以前のコンテキストを基に構築 | **会話を継続** |

### 過去の作業の参照

**NG**: 会話全体をコピー&ペースト
**OK**: `@Past Chats` を使用（必要なコンテキストを選択的にロード）

## 4. エージェント機能の拡張

### Rules（静的ルール）

**場所**: `.cursor/rules/` または `CLAUDE.md`

**含めるべき内容**:
- 必須コマンド（ビルド、テスト）
- コーディングパターン
- 正規のコード例

**避けるべき内容**:
- 完全なスタイルガイド（代わりにリンターを使用）
- 過剰な指示

**原則**: シンプルに始める。エージェントが同じミスを繰り返した時だけ追加する。

### Skills（動的スキル）

オンデマンドでロードされる機能：

| 種類 | 説明 | 例 |
|------|------|-----|
| カスタムコマンド | `/` でトリガー | `/pr`, `/fix-issue` |
| Hooks | アクション前後に実行されるスクリプト | テスト自動実行 |
| ドメイン知識 | 特定分野の専門知識 | プロジェクト固有のルール |

## 5. 長時間ループパターン

### Hooks を使った反復実行

`.cursor/hooks.json` で設定：

```json
{
  "stop": {
    "command": "node check-completion.js",
    "returns": {
      "followup_message": "継続指示（空なら停止）"
    }
  }
}
```

### 適用シナリオ

- **テスト駆動ワークフロー**: すべてのテストがパスするまで反復
- **デザイン反復**: 視覚的な目標達成まで継続
- **検証可能なゴール**: 明確な成功基準がある作業

## 7. 一般的なワークフロー

### テスト駆動開発 (TDD)

```
1. テスト作成を依頼
2. テストが失敗することを確認
3. コミット
4. 実装を依頼
5. すべてのテストがパスするまで反復
```

### コードベース理解

エージェントをチームメイトのように活用：

- 「ロギングパターンはどうなっている？」
- 「APIエンドポイントはどう作る？」
- 「このエッジケースはどう処理されている？」
- 「この設計判断の理由は？」

### Git自動化

カスタムコマンドの例：

| コマンド | 動作 |
|----------|------|
| `/pr` | コミット → プッシュ → PR作成 |
| `/fix-issue [番号]` | Issue取得 → 修正 → PR |
| `/review` | リンター実行 → 問題報告 |

## 8. コードレビュー戦略

### 生成中のレビュー

- diffビューアを監視
- 方向性が間違っていたら `Escape` で停止

### エージェントレビュー

「Find Issues」をクリック → 行ごとにバグを分析

### 自動レビュー（Bugbot等）

- PRの自動レビュー
- 問題検出と改善提案

### アーキテクチャ図

実装前にMermaid図をリクエスト：

```
「この機能のアーキテクチャをMermaid図で示して」
→ 設計を検証
→ 承認後に実装開始
```

## 9. 並列実行

### Git worktree サポート

- 各エージェントに隔離された環境を自動作成
- ファイル競合を防止
- 各エージェントが独立して編集・ビルド・テスト

### マルチモデル比較

1. 同じプロンプトを複数モデルに送信
2. 結果を並べて比較
3. 最適な解決策を採用

## 11. デバッグモード

### 手順

1. **仮説を立てる**: 複数の可能性を検討
2. **計測**: ロギングステートメントを挿入
3. **再現**: ランタイムデータを収集
4. **分析**: 実際の動作を特定
5. **修正**: 証拠に基づいた対処

### 効果的なケース

- レースコンディション
- タイミング問題
- パフォーマンス問題
- リグレッション

## 12. ワークフロー成熟のベストプラクティス

### 具体的なプロンプトを書く

**NG（曖昧）**:
```
ログアウトのテストを書いて
```

**OK（具体的）**:
```
ログアウトのテストを書いて。
- エッジケースをカバー
- `__tests__/` のパターンに従う
- モックは使わない
```

### セットアップは慎重に反復

1. シンプルに始める
2. 繰り返しミスを観察した時だけルールを追加
3. 過剰な指示は逆効果

### 徹底的にレビュー

> AI生成コードは正しく見えても、微妙なエラーを含むことがある

### 検証可能なゴールを提供

エージェントに明確な成功基準を与える：

- 型付き言語を使用
- リンターを設定
- テストを書く

### エージェントを有能な協力者として扱う

- 計画を求める
- 説明を要求する
- アプローチに（丁寧に）異議を唱える

## 13. 実践チェックリスト

### タスク開始前

- [ ] **既存コードを確認した**（→ [Issue #001](issues/001-check-existing-code-first.md)）
- [ ] **ユーザー要望を明確にした**
- [ ] **既存コードとユーザー要望の両立方法を考えた**（→ [Issue #003](issues/003-balance-existing-code-and-user-requirements.md)）
- [ ] プランニングモードで計画を立てた
- [ ] 関連ファイルを調査した
- [ ] 不明点を質問した
- [ ] 成功基準を明確にした

> **重要**: 「既存コードに合わせる」は手段であって目的ではない。
> 目的は「ユーザー要望を満たしつつ、既存システムと統合する」こと。

### 実装中

- [ ] 小さなステップで進めている
- [ ] 各ステップでテストを実行している
- [ ] diffを監視している
- [ ] 方向性が間違っていたら停止している

### 完了後

- [ ] すべてのテストがパスした
- [ ] **テスト自体の前提が正しいか確認した**（→ [Issue #002](issues/002-tests-guarantee-consistency-not-correctness.md)）
- [ ] コードをレビューした
- [ ] 型エラーがない
- [ ] リンターエラーがない
- [ ] コミットメッセージが適切

> **注意**: テストが通ることは「正しさ」の証明ではない。
> テストは「一貫性」を保証するが、「その仕様が正しいか」は人間が判断する。

## 14. キーポイント要約

> 「エージェントは急速に進化している。新しいモデルでパターンは変わるが、このガイドは今日のコーディングエージェントで生産性を向上させるはずだ」

**成功の鍵**:
- エージェントの能力
- 人間の判断
- 明確な計画
- 体系的なフィードバック

これらを組み合わせることで最大の効果を発揮する。

## 15. コード設計の基本原則

> 2026-01-17追加: エージェントが見落としがちな設計ミス

### 遅延処理（Lazy Evaluation）

**NG: 全データを先に処理**
```javascript
// 全画像をbase64化してから解析
const allImages = await processAllImages(paths);  // 遅い
const results = await analyze(allImages);  // base64を使わない
```

**OK: 必要な時に必要な分だけ**
```javascript
// 解析に必要なのはファイルパスだけ
const results = await analyze(paths);
// 出力時に必要な画像だけbase64化
const output = await processForOutput(results);
```

### 共通コードの抽出

**NG: コピペ**
```javascript
// file1.ts
const inputs = paths.map(p => ({ fileName: basename(p), path: p }));

// file2.ts
const inputs = paths.map(p => ({ fileName: basename(p), path: p }));

// file3.ts
const inputs = paths.map(p => ({ fileName: basename(p), path: p }));
```

**OK: 共通関数化**
```javascript
// shared.ts
export function createInputs(paths) {
  return paths.map(p => ({ fileName: basename(p), path: p }));
}

// file1.ts, file2.ts, file3.ts
const inputs = createInputs(paths);
```

### データフローの確認

実装前に問う：
1. **入力は何か？** → ファイルパス
2. **出力は何か？** → 解析結果JSON
3. **各ステップで本当に必要なデータは？** → パスだけで十分
4. **処理結果は次で使われるか？** → base64は解析に不要

### チェックリスト

- [ ] 処理した結果を次のステップで使っているか？
- [ ] 同じコードが複数箇所にないか？
- [ ] 必要になるまで処理を遅延できないか？
- [ ] データの流れを図示できるか？
