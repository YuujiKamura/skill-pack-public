# AI連携

会話数: 8 | パターン数: 5

## 1. 言語非依存プロンプト仕様 (SSOT) への統合

CLI (Rust) と Web (TypeScript/React) など、異なる実行環境間でAIの振る舞いを一致させるため、プロンプトやパラメータを `prompt-spec.json` などの共通ファイルで一元管理する。ハードコードを排除し、型安全な方法で各言語から参照する。

### 手順

1. 既存の各プロジェクトにおけるプロンプトのハードコード箇所を特定する
2. 言語非依存の `prompt-spec.json` を作成し、共通の命令文や計算ロジック、期待する出力構造を定義する
3. Rust側では `include_str!` を、TS側では JSON インポートを使用して、定義した仕様をコードから参照するよう書き換える
4. プロンプトの変更を反映した後、各プロジェクトでビルドおよびテスト（`npx tsc`, `cargo test`等）を実行し、一貫性を検証する

出現頻度: 3回

## 2. コンタクトシート方式によるVision AIの精度改善

多数の画像を個別のファイルとしてAIに渡すとアテンションが分散し、精度が低下する。これを解決するため、複数の画像を番号付きの「コンタクトシート（1枚の大きな画像）」に合成し、空間的な位置関係を維持したまま1回のコールで解析させる。

### 手順

1. 解析対象の画像群（Before/Afterなど）をグリッド状に配置したコンタクトシート画像を生成する
2. 各画像に「番地」や「番号」を付与し、AIが特定の対象を指し示せるようにする
3. 「グリッド上の番号同士のペアリング」のように、画像内の空間的な対応関係を問うプロンプトを作成する
4. 1回のAPIコールで全てのペアリングや解析結果を抽出する

出現頻度: 1回

## 3. プロジェクト横断的な「メモリ」と知見の永続化

セッションの中断や再起動、あるいは複数プロジェクト間での作業において、調査した内容や「精度の傾向」をMarkdownファイル（`MEMORY.md` や `accuracy-findings.md`）として記録し、AIがコンテキストを継続できるようにする。

### 手順

1. プロジェクトのルートや `.claude/projects/` 下に知見記録用のMarkdownファイルを作成する
2. 「3枚なら精度100%だが21枚だと10%になる」といった、試行錯誤から得られた具体的な定量的知見を追記する
3. 新しいセッションや別のプロジェクトの作業を開始する際、まずこれらのメモリファイルを読み込み、過去の失敗や成功パターンを把握する

出現頻度: 2回

## 4. ドメイン知識に基づいた手書き図面の構造化解析

測量図などの専門的な手書き図面を解析する際、単に文字を読み取るのではなく、図面の構造（「ハブ点からの放射状分割」など）をAIに理解させ、目的のデータ構造（CSV等）に直接マッピングする。

### 手順

1. 入力画像（手書き図面）の幾何学的なルール（基点、方向、接続関係）を特定する
2. 出力先となるデータ構造（`trianglelist` のCSV定義など）をコードや既存ファイルから解析する
3. 「左から順に」「ハブ点から放射状に」といった、人間が図面を読む際のアルゴリズムをプロンプトで指示する
4. 抽出した数値を定義済みのデータ型に合わせて整形し、流し込める形式で出力する

出現頻度: 1回

## 5. GitHub/MCP等を活用した自律的な外部ツール調査

未知のライブラリやツール（CocoIndexなど）を導入する際、GitHub API (`gh api`) や Web検索を自律的に使用して、リポジトリ構造、サンプルコード、最新のトレンドを調査し、プロジェクトへの適用可能性を判断する。

### 手順

1. 対象ツールのリポジトリURLから、`gh repo view` や `gh api` を用いて構成ファイルやディレクトリ一覧を取得する
2. `examples` フォルダなどから実装サンプルを特定し、その内容（`main.py` 等）を読み解く
3. 調査結果に基づき、自プロジェクトの構成（`pyproject.toml` や `.env`）に合わせてインテグレーションプランを策定し実行する

出現頻度: 1回
