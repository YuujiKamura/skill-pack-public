# 作業時の基本原則

## 1. 日本語ファイルのエンコーディング

**日本語WindowsのExcelで開くCSVはcp932（Shift-JIS）を使う**

```python
# NG: UTF-8（Excelで文字化け）
df.to_csv('output.csv', encoding='utf-8')

# OK: cp932
df.to_csv('output.csv', encoding='cp932')
```

UTF-8で保存すると、WindowsのExcelで開いた時に文字化けする。

---

## 2. 創作禁止

**知らないことを推測で説明しない**

- 製品名・専門用語など、確実に知らないものは「知らない」と言う
- または、ファイルや資料を確認して実際の情報を提示する
- 「おそらく〜だと思います」という推測での説明は禁止

---

## 3. 複数資料間の整合性確認

**どれが正解かではなく、不整合を発見することを出発点にする**

例：面積展開図と切削計算書で数値が異なる場合
- NG: 「面積展開図が正しいです」と断定
- OK: 「面積展開図では1777.57㎡、切削計算書では1777.92㎡。0.35㎡の差異があります。原因は丸め処理の違いと思われます」

不整合の原因を特定し、ユーザーに判断材料を提示する。

---

## 4. 数式範囲の確認

**SUMなどの範囲は、含まれるセルの内容を確認してから決定する**

- 既存ファイルの数式を修正する際、範囲が意図通りか確認
- 異なる種類の項目が混在していないかチェック
- 行の挿入・削除で範囲がずれていないか確認

```
# NG: 見た目で範囲を決める
=SUM(J8:J16)  # J8がアスファルトなのに塗料計に含めてしまう

# OK: 項目を確認して個別参照
=J12+J13+J15+J16  # 白塗料の行だけを明示的に指定
```

---

## 5. 表計算は「書式込みで書く」が前提

**表計算ソフトでは、値と書式はセットで一つのデータ。値だけ書くのは半分しか書いていない**

セルには値の裏に書式情報が常にある:
- 数値書式（`#,##0_ ` / `#,##0.0_ ` / `#,##0.00_ `）
- 罫線パターン（hairline / thin / dot）
- フォント色（黒 / 青 `#244062` / 赤 `#ff0000`）
- 配置（center / left）
- コード列、カテゴリ行、複数行構造

この前提が染み込んでいれば、既存の表に行を追加するときに**既存行をコピーして値だけ変える**のが自然な初手になる。書式をゼロから組み立てるという発想自体がおかしい。

### 手順

```
1. 同じ構造の既存行を特定する
2. 行ごとコピーして挿入先にペースト（書式が全部ついてくる）
3. 値だけ書き換える
4. 不要な値をクリア
```

### VBAでの実装

```vba
ws.Rows("11:12").Copy Destination:=ws.Rows("57")
ws.Range("E57").Value = "交通誘導警備員"
ws.Range("H57").Value = "交通誘導員B"
```

### やってはいけないこと

- ❌ edit_cellsで値だけ入れて、あとから書式を合わせようとする
- ❌ 書式の差分を1つずつ調べて set_cell_formats で再現する
- 値と書式を別々に扱っている時点で、表計算ソフトの使い方を間違えている
