# スキル管理ルール

## YAML frontmatter書式（必須）

```yaml
---
name: skill-name
description: 説明文。キーワード1、キーワード2と言われた時に使用。
---
```

**NG:** Markdownだけで書く（`# skill-name`から始める）
**OK:** YAML frontmatterで始める（`---`で囲む）

---

## 説明文の書き方（番号付きユースケース）

公式パターンに従う：

```yaml
description: 概要。(1) ユースケース1、(2) ユースケース2、(3) ユースケース3、(4) ユースケース4、(5) ユースケース5。キーワード1、キーワード2、キーワード3と言われた時に使用。
```

**例:**
```yaml
# NG: 短すぎる
description: Gmail APIでメール作成。

# OK: ユースケース + キーワード
description: Gmail API操作。(1) メール下書き作成・送信、(2) 受信メール検索・フィルタリング、(3) 添付ファイル付きメール作成、(4) ラベル管理・スレッド取得、(5) 返信・転送の自動化。Gmail、メール、下書き、送信、検索、添付、返信、転送、受信トレイ、ラベル、連絡、通知と言われた時に使用。
```

---

## キーワード選定のコツ

1. **ユーザーの語彙に合わせる**: 技術用語だけでなく日常語も
   - NG: `IMPORTRANGE` だけ
   - OK: `IMPORTRANGE、参照、データ連携、一覧表`

2. **類義語を網羅する**:
   - `下請け、協力会社、業者`
   - `メール、連絡、通知`

3. **動詞も入れる**:
   - `作成、追加、更新、修正、改善`

4. **よく発火しなかったキーワードを追記していく**:
   - 使用頻度が高いスキルほどキーワードが洗練される

## 保存先ルール

**スキルは `<CLAUDE_HOME>/skills/` に保存する**

スキルはキーワードに反応して自動的に読み込まれるもの。プロジェクト固有のフォルダに置くと、他のプロジェクトで使えなくなる。

## 保存先

```
<CLAUDE_HOME>/skills/
├── skill-management/
├── weekly-report/
├── 3week-schedule-chart/
├── construction-initial-tasks/
└── ...
```

## スキル作成手順

```bash
mkdir -p <CLAUDE_HOME>/skills/<skill-name>
# SKILL.md を作成
```

## 例外

チーム共有が必要な場合のみ、プロジェクトの `.claude/skills/` に置く。

## コード変更時のルール

**ファイルを変更したら、ヘッダーに変更履歴を追記する**

```rust
//! モジュール名
//!
//! ## 変更履歴
//! - 2026-01-02: 機能追加の説明
```

```javascript
/**
 * ファイル名
 *
 * ■ 変更履歴
 * - 2026-01-02: 機能追加の説明
 */
```

これを忘れると何をしたか分からなくなる。
