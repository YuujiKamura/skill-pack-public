# DXF/CAD

会話数: 15 | パターン数: 5

## 1. DXFファイル構造の完全化とCAD互換性修正

内製ビューアで表示できても標準的なCAD（AutoCAD等）で開けない問題を解決するため、DXFのフルセクション構造を実装し、仕様の不備を修正するパターン。

### 手順

1. HEADER, TABLES, BLOCKS, ENTITIES, OBJECTS, EOFの全セクションを順序通り実装
2. 各エンティティへのハンドル(group 5)とオーナー参照(group 330)の付与
3. ヘッダー先頭の不正な空行削除やINSUNITS定数の整合性確認
4. 実績のあるappモジュールの出力ロジックをcommonモジュールへ共通化

出現頻度: 5回

## 2. ezdxfライブラリによる自動監査(audit)ワークフロー

生成されたDXFがCAD仕様に準拠しているかをPythonの外部ツールを用いて機械的に検証し、エラー箇所を特定する手順。

### 手順

1. 環境内でのPython/ezdxfのインストール確認
2. `ezdxf audit`コマンドによる構造不整合や参照エラーの検出
3. `ezdxf.recover`による破損ファイルの読み込み試行とエラーログ解析

出現頻度: 4回

## 3. 描画中間表現(Drawing IR)による表示・出力の同期

UI上のプレビュー（egui等）とDXFエクスポートのロジック乖離を防ぐため、プリミティブな描画データ（Line, Text, Circle等）を介して出力を一元化するパターン。

### 手順

1. 汎用的な描画コマンドを保持するDrawing IR構造体の定義
2. IRからUIフレームワーク（egui等）への描画変換アダプタの実装
3. IRからDXF Writerへの出力変換アダプタの実装
4. 共通のスタイリング（SectionStyle等）の適用

出現頻度: 4回

## 4. 肥大化したレイアウト計算ロジックのモジュール分割

横断図やグリッド配置など、1000行を超える巨大な計算ファイルを、責務ごとに分割してコンテキスト制限や保守性を改善するリファクタリングパターン。

### 手順

1. 計算パラメータ（Params）、境界計算（Bounds）、描画実行（Render）へのファイル分割
2. 重複する列幅・オフセット計算ロジックの共通関数化
3. ビルダーパターンの導入による複雑な配置設定の整理
4. ドメイン固有ロジックと汎用描画ロジックの分離

出現頻度: 6回

## 5. ドメインモデル準拠のCSV→DXF変換CLI構築

現場でのCSVデータをアプリを起動せずにCADで確認するため、実績のある計算モデル（Triangle/Dims等）を用いてCLIから直接DXFを生成するパターン。

### 手順

1. Androidアプリ等で実績のある計算モデルを共通モジュールへ抽出
2. 独自の配置計算（placeTriangles等）をモデル側の計算メソッド（calcPoints等）へ置き換え
3. desktopモジュールにCSV入力・DXF出力を繋ぐCLIメインクラスを実装

出現頻度: 3回
