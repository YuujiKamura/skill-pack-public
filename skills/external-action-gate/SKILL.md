# 外部アクションゲート

外部に影響するアクションを実行する前に、必ず停止してユーザーの明示的許可を取れ。

## なぜこれが必要か

AIエージェントは「質問するな、自分で判断しろ」という指示を受けている。しかしこれは**ローカルの作業**に対する指示であり、**外部への発信**には適用されない。コードを書く・テストを走らせる・ファイルを編集するのは取り消せる。PRを送る・Issueを立てる・コメントを投稿するのは取り消せない（閉じても通知と履歴が残る）。

過去の失敗: psmux本家に品質の低いPRを無断送信。互換性破壊・unsafe直書き・テストなしの状態で。閉じたが履歴は残った。

## 対象アクション

以下のいずれかを**実行しようとした時点**で停止:

- `gh pr create` — PRの作成（どのリポジトリ宛でも）
- `gh issue create` — Issueの作成（自分のリポジトリでも確認）
- `gh pr comment` / `gh issue comment` — コメント投稿
- `git push` — リモートへのpush（自分のフォークでも、upstreamに見える）
- 外部サービスへの投稿（Slack、メール等のAPI呼び出し）

## 停止時にやること

1. **何をしようとしているか**を1行で述べる
2. **対象リポジトリ/サービス**を明示する（自分のフォークか、他人のリポジトリか）
3. **ユーザーの明示的な「やれ」「送れ」「OK」等の許可**を待つ
4. 許可がなければ実行しない。沈黙は拒否と同じ

## 判断基準

「自分で判断して実行しろ」が適用されるのは:
- ファイルの読み書き
- ビルド・テスト実行
- ローカルのgit操作（commit、branch、stash）

「確認を取れ」が適用されるのは:
- 外部に見えるすべてのアクション
- 取り消しても痕跡が残るアクション

迷ったら聞け。外部アクションに関しては「聞くな」ルールは適用されない。

## PRを出す前の品質ゲート

`gh pr create`の許可を求める前に、以下を全て満たしているか自己チェックしろ。満たしていなければ「まだ出せる状態ではない、残りの作業はこれ」と報告しろ。

### 設計
- 互換性を壊していないか（ファイルフォーマット、API、設定の既存動作）
- unsafeを本当に避けられないか確認したか（標準ライブラリやコマンド呼び出しで代替できないか）
- 追加の依存は最小限か
- 本家の既存コードパターンに合わせているか（同じ問題を解いてる別の箇所を参考にしたか）

### テスト
- 修正したバグの再現テストがあるか
- 既存テストが全パスするか
- テストが並列実行で安定するか（3回連続全パス）
- テストが共有状態（ファイルシステム、環境変数）に依存していないか

### 履歴
- 手戻りのゴミコミットが残っていないか（squash済みか）
- コミットメッセージが修正内容を正確に説明しているか
- upstreamの最終コミットの上に綺麗に乗っているか

### プロセス
- 本家にIssueで設計を先に提案したか（PRの前にIssue）
- 「避けられない」「不可能」と言ってる部分は本当か（嘘ついてないか検証したか）
