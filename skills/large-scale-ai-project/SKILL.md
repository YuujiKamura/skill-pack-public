# 大規模AIコーディングプロジェクトのベストプラクティス

FastRender（86万行Rust）の知見に基づく。

## 1. 90/10ルール

**90%を主目標、10%をインフラに配分**

```
❌ 悪い指示
「コードを改善して」
「きれいにして」
「最適化して」

✅ 良い指示
「この関数のレスポンス時間を100ms以下にして」
「このテストが通るように実装して」
「エラー時にユーザーに具体的なメッセージを表示して」
```

曖昧な指示 → AIが迷走 → 時間の無駄

## 2. シングルステップループ

**1つずつ実行 → 比較 → 修正 → 検証**

```
❌ バッチ処理
「AとBとCを全部直して」
→ どこで壊れたか分からない

✅ シングルステップ
1. Aを直す → テスト → OK
2. Bを直す → テスト → 失敗 → 原因特定 → 修正
3. Cを直す → テスト → OK
```

## 3. ワークストリーム分離

並列エージェント使用時、**作業領域を明確に分離**：

```
ワークストリーム例（FastRenderの場合）:
├── rendering/     # エージェントA担当
├── parsing/       # エージェントB担当
├── optimization/  # エージェントC担当
└── testing/       # エージェントD担当
```

**分離のルール:**
- ファイルの所有権を明確に
- 共有部分は1つのエージェントが管理
- インターフェースを先に定義

## 4. 優先度の明確化（P0-P5）

| 優先度 | 内容 | 対応 |
|--------|------|------|
| P0 | クラッシュ、データ損失 | 即座に対応 |
| P1 | 主機能が動かない | 当日中 |
| P2 | 機能の一部が動かない | 今週中 |
| P3 | パフォーマンス問題 | 計画的に |
| P4 | 改善要望 | バックログ |
| P5 | 新機能アイデア | 検討 |

AIに優先度を伝えないと、P5に時間を使ってP0を放置する。

## 5. リソース制限の厳格適用

**すべてが誤動作する可能性を想定**

```bash
# タイムアウト必須（SIGKILLフォールバック）
timeout -k 10s 60s ./heavy-process

# メモリ制限
ulimit -v 4000000  # 4GB制限

# 並列数制限
parallel -j 4  # 最大4並列
```

AIが無限ループ/暴走するリスクを常に考慮。

## 6. 試行錯誤パターン

**失敗しやすいアプローチ:**
- 設定だけで解決しようとする
- 後処理で修正しようとする
- 一度に大きく変更する

**成功しやすいアプローチ:**
- パイプラインの順序を守る
- 早期にデータを準備する
- 小さく変更してすぐ検証
